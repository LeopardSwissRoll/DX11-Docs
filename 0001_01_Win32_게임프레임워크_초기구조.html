CONCEPT 01
PeekMessage 기반 게임 루프
일반 Windows 앱은 GetMessage()로 메시지를 기다리는데, 이건 메시지가 올 때까지 블로킹(멈춤)한다. 게임은 매 프레임 계속 돌아야 하니까 PeekMessage()를 써서 논블로킹으로 처리한다.
GetMessage — 블로킹

메시지 없으면 멈춤. 일반 앱(메모장 등)에 적합.

PeekMessage — 논블로킹

메시지 없어도 바로 리턴. 게임 루프에 적합.

C++ PeekMessage 게임 루프 Client.cpp
while (true)
{
    if (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE))
    {
        // 메시지가 있다
        if (msg.message == WM_QUIT)
            break;

        // 단축키 관련 내용이면 TranslateAccelerator 처리
        if (!TranslateAccelerator(msg.hwnd, hAccelTable, &msg))
        {
            TranslateMessage(&msg);
            DispatchMessage(&msg);   // → WndProc 호출
        }
    }
    else
    {
        // 윈도우 데드타임 → 게임 로직!
        // Logic();
    }
}
🏪 ANALOGY — 편의점 알바 비유
PeekMessage = 손님 있나 힐끗 보고, 없으면 진열대 정리(게임 로직) 하는 것.
GetMessage = 손님 올 때까지 카운터에서 멍하니 기다리는 것.

게임은 1초에 60번 이상 화면을 그려야 하니까, 멍하니 기다리면 안 된다!
CONCEPT 02
CRTP 싱글톤 패턴
CRTP (Curiously Recurring Template Pattern) — 자식 클래스가 자기 자신을 템플릿 인자로 넘기는 패턴. 싱글톤(전역 인스턴스 하나)을 만들 때 상속만으로 쉽게 적용할 수 있다.
C++ CRTP Singleton Singleton.h
template<typename T>
class Singleton
{
public:
    static T& Instance() {
        static T Inst;      // 프로그램 전체에서 딱 하나
        return Inst;
    }
    // 복사/이동 생성자, 대입 연산자 삭제 → 인스턴스 하나 보장
};

// 사용법: 자기 자신을 넘긴다!
class ObjectPool : public Singleton<ObjectPool> { ... };
WHY CRTP?
Singleton<ObjectPool>::Instance()가 ObjectPool&를 반환하도록 컴파일 타임에 타입이 결정되니까. 캐스팅 없이 바로 쓸 수 있어서 편하고 안전하다.
CONCEPT 03
Object 기본 클래스
게임 객체(플레이어, 몬스터, 총알 등)가 공통으로 가질 속성을 정의한 추상 클래스.
C++ 핵심 멤버 변수 Object.h
class Object
{
    friend class ObjectPool;

private:
    bool        _bEnable = false;    // 활성 상태? (살아있지만 일시 비활성 가능)
    bool        _bActive = false;    // 살아있나? (false면 삭제 대상)
    bool        _IsPooled = false;   // 풀에서 빌려온 객체인가?
    eObjectType _Type = eObjectType::END;  // 객체 종류

public:
    virtual void Destroy() = 0;  // 순수 가상 → 자식이 반드시 구현
};
ACTIVE vs ENABLE
Active = 존재 여부 (false면 오브젝트 자체가 사라짐)
Enable = 활성 여부 (살아있지만 일시적으로 멈출 수 있음 — 예: 빙결 상태)
CONCEPT 04
ObjectPool — 객체 재사용 시스템
게임에서 총알, 이펙트 같은 건 수백 개씩 생겼다 사라진다. 매번 new/delete 하면 느려지니까, 다 쓴 객체를 삭제하지 않고 풀에 반환해서 재사용한다.
C++ 풀 구조 ObjectPool.h
// 타입별로 큐를 가진 맵
std::map<eObjectType, std::queue<Object*>> _Pools;

// 빌리기: 풀에 있으면 꺼내고, 없으면 new
T* Rent() { ... }

// 반환: delete 하지 않고 풀에 다시 넣기
void Return(Object* obj) { ... }
🎳 ANALOGY — 볼링장 비유
볼링공을 한 번 던질 때마다 새 공을 사고, 던진 후 부수는 건 미친 짓이잖아?
던진 공이 레일을 타고 돌아오면 다시 집어서 던지면 되는 거야.

Rent() = 공을 집는 것 / Return() = 공이 레일로 돌아오는 것
