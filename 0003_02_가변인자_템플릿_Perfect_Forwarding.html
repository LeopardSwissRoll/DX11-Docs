가변 인자 템플릿 +
Perfect Forwarding

New<TestObject>(1, "hello") 한 줄이 어떻게 작동하는지. ...이 뭔지, std::forward가 없으면 뭐가 달라지는지.

출발점: Defines.h의 New 함수

C++ 이 4줄에 고급 문법 3개 Core/Defines.h
template<typename T, typename ...Args>
Ptr<T> New(Args&&... args)
{
    return new T(std::forward<Args>(args)...);
}
이 4줄 안의 3가지 고급 문법
1
typename ...Args — 가변 인자 템플릿 (Parameter Pack). 인자 개수 제한 없음.
2
Args&& — 전달 참조 (Forwarding Reference). lvalue/rvalue 모두 받음.
3
std::forward<Args>(args)... — Perfect Forwarding. 인자의 원래 종류를 보존.

개념 1: 가변 인자 템플릿 — typename ...Args

문제: 인자 개수가 제각각

C++ 인자 0개, 1개, 2개...
// 인자 0개
Ptr<TestObject> a = New<TestObject>();

// 인자 1개
Ptr<TestObject> b = New<TestObject>(42);

// 인자 2개
Ptr<Monster> c = New<Monster>("고블린", 100);

이걸 하나의 함수로 어떻게 처리하지?

해결: Parameter Pack

C++ ... 의 의미
template<typename T, typename ...Args>
//                    ^^^^^^^^^^^^
//                    "0개 이상의 타입을 묶음으로 받겠다"

Ptr<T> New(Args&&... args)
//         ^^^^^^^^^^^^^
//         "묶음에 들어온 인자들을 그대로 나열"
  • typename ...Args — 타입 팩. 인자의 타입을 0개 이상 아무렇게나 받음.
  • Args&&... args — 인자 팩. 타입 팩에 대응하는 실제 값들.
  • ... — 팩 전개(expansion). 컴파일러가 인자 개수에 맞게 코드를 생성함.

컴파일러가 실제로 하는 일

New<TestObject>(42, "hello") 호출 시 팩 전개
1
Args = {int, const char*}
2
args = {42, "hello"}
3
전개: new TestObject(std::forward<int>(42), std::forward<const char*>("hello"))

인자가 3개면 3개로, 0개면 0개로 — 컴파일 타임에 맞는 코드가 생성됨.

개념 2: lvalue vs rvalue

Perfect Forwarding을 이해하려면 이 구분이 필수:

lvalue — "주소를 가진 것"
C++
int x = 10;        // x는 lvalue — &x 가능
TestObject obj;    // obj는 lvalue
rvalue — "임시 값"
C++
42                 // 이름 없는 임시 값
std::string("hi")  // 이름 없는 임시 객체
std::move(obj)     // "더 안 쓸 거야" 선언

왜 중요한가?

C++ 복사 vs 이동
class BigObject {
    BigObject(const BigObject& other) { /* 복사: 느림, 메모리 할당 */ }
    BigObject(BigObject&& other)      { /* 이동: 빠름, 포인터만 옮김 */ }
};
lvalue 넘기면

복사 생성자 호출 → 느림
데이터를 통째로 복제

rvalue 넘기면

이동 생성자 호출 → 빠름
포인터만 옮기고 원본은 비움

📦 ANALOGY — 이사 비유
복사(lvalue) = 집에 있는 물건을 하나하나 똑같이 새로 사서 새 집에 배치. 비쌈.
이동(rvalue) = 이삿짐 트럭에 그대로 실어서 옮기기. 원래 집은 텅 비지만 빠름.

std::forward는 "이 물건이 원래 옮길 물건(rvalue)이었으면, 복사하지 말고 트럭에 싣자"라고 알려주는 역할.

개념 3: 전달 참조 — Args&&

C++ 전달 참조 (Universal Reference)
template<typename T>
void foo(T&& arg)  // ← 이게 전달 참조

T&&가 템플릿 매개변수에 붙으면 특별한 의미가 있어:

넘긴 것T 추론T&& 결과
lvalue int xT = int&int& && → int& (lvalue 참조)
rvalue 42T = intint&& (rvalue 참조)
참조 축약 (Reference Collapsing)
& && → & / && && → &&
이 규칙 덕분에 T&&는 lvalue든 rvalue든 다 받을 수 있는 만능 참조가 됨.

개념 4: std::forward — "받은 그대로 넘겨라"

forward 없이 (변질됨)
C++
template<typename T, typename ...Args>
Ptr<T> New_Bad(Args&&... args)
{
    return new T(args...); // forward 없음!
}

args는 이름이 생겼으니 lvalue → rvalue가 복사로 변질

forward 있으면 (원본 유지)
C++
template<typename T, typename ...Args>
Ptr<T> New(Args&&... args)
{
    return new T(std::forward<Args>(args)...);
}

rvalue는 rvalue로 유지 → 이동 생성자 호출 가능!

전체 흐름 추적: New<TestObject>(42)

New<TestObject>(42) │ ├─ T = TestObject, Args = {int}, args = {42} │ ├─ #ifdef _USE_MEMORY_POOL: │ new TestObject(std::forward<int>(42)); │ ├─ Object::operator new(sizeof(TestObject)) ← MemoryPool │ └─ TestObject(42) 생성자 호출 │ → TestObject* 반환 (= Ptr<TestObject> = T*) │ ├─ #elif _USE_OBJECT_POOL: │ ObjectPool::Instance().Rent<TestObject>(); │ ⚠ 인자(42)가 무시됨! (Rent는 인자 안 받음) │ → TestObject* 반환 │ └─ #else (shared_ptr): std::make_shared<TestObject>(std::forward<int>(42)); ├─ 힙에 TestObject + control block 할당 └─ TestObject(42) 생성자 호출 → shared_ptr<TestObject> 반환
ObjectPool 전략의 한계
_USE_OBJECT_POOL일 때 New<T>(args...)의 인자가 무시됨. Rent<T>()는 이미 생성된 객체를 큐에서 꺼내는 것이지, 새로 만드는 게 아니니까. 생성자 인자 전달이 필요한 경우 ObjectPool 전략에서는 별도 초기화 함수가 필요.
