RTTI와 type_index —
타입을 키로 쓴다는 것

ObjectPool의 키가 eObjectType enum에서 std::type_index로 바뀌었다. 이게 뭔지, 왜 바꿨는지, 성능은 괜찮은지.

변경 전후 비교

이전 (v2) — enum 키
map<eObjectType, queue<Object*>>

새 타입 추가 → enum 수정 필요
Rent(eObjectType) → 타입을 2번 명시
현재 (v3) — type_index 키
map<std::type_index, queue<Object*>>

새 타입 추가 → 아무것도 안 해도 됨
Rent<T>() → 타입 1번만
C++ 이전 — enum 키 v2 ObjectPool.h
// Info.h
enum class eObjectType
{
    OBJECT_TYPE_TEST_OBJ,
    OBJECT_TYPE_MONSTER,  // ← 새 타입마다 추가 필요
    END
};

// ObjectPool.h
std::map<eObjectType, std::queue<Object*>> _Pools;

// Rent
auto pool = _Pools.find(type);  // eObjectType을 인자로 받아야 함
C++ 현재 — type_index 키 v3 Core/ObjectPool.h
#include <typeindex>

std::map<std::type_index, std::queue<Object*>> _Pools;

// Rent
auto pool = _Pools.find(std::type_index(typeid(T)));

// Return
_Pools[std::type_index(typeid(*obj))].push(obj);

RTTI가 뭔가?

RTTI = Run-Time Type Information (런타임 타입 정보)

C++은 기본적으로 컴파일 타임에 타입을 결정해. 근데 다형성(상속 + virtual)을 쓰면 런타임에 "이 포인터가 실제로 뭘 가리키고 있는지" 알아야 할 때가 있어.

C++ 컴파일 타임 vs 런타임 타입
Object* obj = new TestObject();  // 컴파일 타임 타입: Object*
                                 // 런타임 실제 타입: TestObject

typeid(*obj)  // → TestObject의 type_info를 반환 (런타임에 판단!)
RTTI — 포인터 뒤의 진짜 타입을 알아내는 것
Object* obj →
→
컴파일러가 아는 것: Object 실제: TestObject typeid(*obj) → TestObject
RTTI를 지원하는 C++ 기능들
typeid() — 타입 정보 얻기
dynamic_cast<> — 안전한 다운캐스팅
std::type_info — typeid의 반환 타입
std::type_index — type_info를 컨테이너 키로 쓸 수 있게 래핑

typeid 기초

C++ typeid 사용법
#include <typeinfo>

TestObject test;
Object* ptr = &test;

typeid(test)       // → TestObject의 type_info (정적 타입)
typeid(*ptr)       // → TestObject의 type_info (런타임 타입!)
                   //   ※ Object에 virtual 함수가 있어야 작동

typeid(test).name()  // → "class TestObject" (컴파일러마다 다름)

typeid는 const std::type_info&를 반환하는데, 이걸 직접 map 키로 쓸 수 없어 — 복사가 안 되거든. 그래서 std::type_index로 감싸는 거야.

type_index — typeid를 키로 쓸 수 있게

C++ type_index 기본 사용
#include <typeindex>

std::type_index ti = std::type_index(typeid(TestObject));

// map 키로 사용 가능!
std::map<std::type_index, int> counts;
counts[std::type_index(typeid(TestObject))] = 10;
counts[std::type_index(typeid(Monster))] = 20;

type_index는 내부적으로 type_info*를 들고 있고, <, ==, hash 연산을 제공해서 map/unordered_map 키로 쓸 수 있어.

type_info는 직접 키로 못 쓰고, type_index로 감싸야 함
1
typeid(TestObject) → const type_info& 반환 (복사 불가!)
2
std::type_index(typeid(T)) → 래핑. 복사 가능, 비교 가능, hash 가능
3
map<type_index, ...> → 키로 사용 가능!

ObjectPool에서 어떻게 쓰이나

Rent — 컴파일 타임에 T의 타입으로 풀을 찾음

C++ Rent 내부 Core/ObjectPool.h
template<typename T>
T* Rent()
{
    auto pool = _Pools.find(std::type_index(typeid(T)));
    //                                      ^^^^^^^^
    //                      컴파일 타임에 T의 타입 정보를 키로 사용
    if (pool == _Pools.end())
        return nullptr;

    // ... 큐에서 꺼내서 반환
}

Return — 런타임에 실제 객체 타입으로 풀을 찾음

C++ Return 내부 Core/ObjectPool.cpp
void ObjectPool::Return(Object* obj)
{
    // ...
    _Pools[std::type_index(typeid(*obj))].push(obj);
    //                           ^^^^
    //            런타임에 obj의 실제 타입을 확인
    //            Object* 로 받았지만 TestObject인지 Monster인지 알 수 있음
}
typeid(T) vs typeid(*obj) — 컴파일 타임 vs 런타임
Rent<TestObject>()

typeid(T)
T = TestObject (템플릿 인자)
컴파일 타임에 결정됨

Return(Object* obj)

typeid(*obj)
obj가 가리키는 실제 타입
런타임에 결정됨 (vtable 참조)

PreAlloc — 사전 할당

C++ PreAlloc 템플릿 Core/ObjectPool.h
template<typename T>
void PreAlloc(uint32 count)
{
    for (uint32 i = 0; i < count; ++i)
    {
        T* obj = new T();
        _Pools[std::type_index(typeid(T))].push(obj);
    }
}

// 사용:
PreAlloc<TestObject>(10);
// → typeid(TestObject)를 키로 10개의 TestObject를 큐에 넣음

성능 비용 — 게임에서 괜찮은가?

TYPEID 비용
다형 타입 (virtual 함수 있는 클래스): vtable 포인터 한 번 따라감 → 거의 0에 가까운 비용
비다형 타입: 컴파일 타임에 결정 → 비용 없음
TYPE_INDEX 비교 비용
내부적으로 포인터 비교 → int 비교만큼 빠름
map<type_index, ...> 탐색: O(log N) — 타입이 수십 개면 사실상 무시 가능
결론
게임에서 문제 없다. ObjectPool의 Rent/Return은 프레임당 수십~수백 번 정도 호출되는데, typeid/type_index 비용은 그 안에서 측정 불가능할 정도로 작다.

다만 일부 게임 엔진은 RTTI를 비활성화하는 경우가 있다 (컴파일러 옵션 -fno-rtti). 그런 환경에서는 type_index를 못 쓰고, enum이나 커스텀 타입 ID 시스템을 써야 한다. 언리얼 엔진이 대표적 — 자체 StaticClass() 시스템을 쓰지.

RTTI 비활성화 시 대안 — 참고만

C++ 커스텀 타입 ID (RTTI 없이)
// 커스텀 타입 ID (RTTI 없이)
class Object
{
    static int NextTypeID() { static int id = 0; return id++; }

    template<typename T>
    static int TypeID() {
        static int id = NextTypeID();  // 타입마다 고유 ID 자동 생성
        return id;
    }
};

// 사용:
int id = Object::TypeID<TestObject>();  // → 0
int id2 = Object::TypeID<Monster>();    // → 1

현재 프로젝트에서는 RTTI가 켜져 있으니까 type_index가 가장 깔끔한 선택.
