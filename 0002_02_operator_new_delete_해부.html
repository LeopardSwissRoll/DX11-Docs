operator new/delete
완전 해부

new가 하는 일을 가로챈다 — MemoryPool이 어떻게 투명하게 작동하는지 한 줄씩 추적. new TestObject() 한 줄 뒤에서 벌어지는 일을 낱낱이 파헤쳐 보자.

원래 new가 하는 일

C++에서 new는 단순한 키워드가 아니다. 사실 2단계로 작동하는 연산자(operator)다. 대부분의 C++ 입문자들이 "new는 메모리 할당이다"라고만 알고 있는데, 정확히 절반만 맞는 말이다.

new의 2단계 — 할당 + 생성
1
operator new(sizeof(TestObject)) 호출 — 메모리 할당
내부적으로 malloc()을 호출해서 요청한 크기만큼 힙에서 빈 공간을 확보한다.
2
생성자(Constructor) 호출 — 객체 초기화
확보한 메모리 위에 멤버 변수를 세팅하고 객체를 "살아있는 상태"로 만든다.

C++ new의 정체 컴파일러 변환
// 우리가 쓰는 코드
TestObject* obj = new TestObject();

// 컴파일러가 실제로 하는 일 (의사 코드)
void* raw = operator new(sizeof(TestObject));  // 1단계: 메모리 할당
TestObject* obj = static_cast<TestObject*>(raw);
obj->TestObject();                                  // 2단계: 생성자 호출
🏠 ANALOGY — 집 짓기
operator new = 빈 땅을 사는 것 (메모리 확보)
생성자 = 땅 위에 집을 짓는 것 (객체 초기화)

땅만 사면 아직 집이 없고, 집을 지어야 사람(데이터)이 살 수 있다. 우리가 하려는 건 "땅을 사는 방법"을 바꾸는 것이다. 부동산(OS)을 통하지 않고, 미리 사둔 부지(MemoryPool)에서 꺼내 쓰는 것!
핵심 인사이트
C++은 operator new를 오버로딩(재정의)할 수 있게 해준다. 즉, 1단계의 "메모리 할당 방법"을 프로그래머가 직접 바꿀 수 있다. 생성자 호출(2단계)은 컴파일러가 자동으로 해주므로 건드릴 필요 없다.

MemoryPool 버전 operator new

Object 클래스에 operator new를 정의하면, Object를 상속한 모든 객체(TestObject, Monster, Bullet 등)가 new할 때 이 함수를 거치게 된다. 한 줄도 바꾸지 않고!

C++ Object::operator new/delete Object.h (_USE_MEMORY_POOL 버전)
class Object
{
protected:
    FSegment _Segment;  // 반납할 때 필요한 정보 저장

public:
    // ★ 커스텀 operator new
    void* operator new(size_t size)
    {
        // 1) MemoryPool에서 메모리를 빌려온다 (OS한테 안 감!)
        FSegment seg = MemoryPool::Instance().Rent((int)size);

        // 2) 나중에 delete할 때 필요한 정보를 객체에 저장
        Object* obj = (Object*)seg._Data;
        obj->_Segment = seg;

        // 3) 메모리 주소를 반환 → 여기 위에 생성자가 호출됨
        return seg._Data;
    }

    // ★ 커스텀 operator delete
    void operator delete(void* ptr) noexcept
    {
        Object* obj = (Object*)ptr;
        MemoryPool::Instance().Return(obj->_Segment);
    }
};

이게 전부다. operator new와 operator delete 딱 두 개만 재정의하면, 기존의 모든 new TestObject() / delete obj 코드가 자동으로 풀을 경유하게 된다.

size_t size는 어디서 오나?
컴파일러가 new TestObject()를 만나면 sizeof(TestObject)를 계산해서 operator new의 첫 번째 인자로 자동 전달한다. 프로그래머가 직접 넘기는 게 아니다!

왜 Object 클래스에 정의하나?

C++의 클래스 멤버 operator new는 상속된다. Object에 한 번만 정의해두면, Object를 상속하는 모든 클래스 (TestObject, Monster, Bullet...)가 자동으로 이 커스텀 new를 사용하게 된다.

글로벌 operator new 재정의

프로그램의 모든 new가 영향받음.
STL 컨테이너, 문자열 등 풀이 필요 없는 것까지 전부 바뀜. 위험!

클래스 멤버 operator new

해당 클래스(와 자식 클래스)의 new만 영향받음.
게임 오브젝트만 풀을 쓰고, 나머지는 기본 동작. 안전!

operator delete

delete도 new와 마찬가지로 2단계로 작동한다. 순서만 반대다.

delete의 2단계 — 소멸 + 해제
1
소멸자(Destructor) 호출 — 정리(cleanup)
객체가 들고 있던 자원(파일 핸들, 다른 메모리 등)을 정리한다.
2
operator delete(ptr) 호출 — 메모리 반환
기본: free() 호출. 우리 버전: obj->_Segment를 꺼내서 MemoryPool에 반환.
C++ delete의 내부 동작 컴파일러 변환
// 우리가 쓰는 코드
delete obj;

// 컴파일러가 실제로 하는 일 (의사 코드)
obj->~TestObject();                                  // 1단계: 소멸자 호출
Object::operator delete(obj);                       // 2단계: _Segment로 풀에 반납
🏚️ ANALOGY — 집 철거
소멸자 = 집 안의 가구를 다 빼내고 철거하는 것
operator delete = 빈 땅을 반납하는 것

우리 버전에서는 땅을 국가(OS)에 반납하지 않고, 우리 부지(풀)에 되돌려놓는다. 나중에 다시 쓰려고!

전체 흐름 추적

new TestObject() 한 줄이 실행될 때, 코드가 어떤 경로를 따라 내려가는지 전체를 추적해보자.

new TestObject() — 전체 호출 경로
1
new TestObject() 호출
컴파일러가 sizeof(TestObject)를 계산 (예: 40바이트)
2
Object::operator new(40) 호출
TestObject는 Object를 상속하므로 Object의 operator new가 불린다
3
MemoryPool::Instance().Rent(40) 호출
싱글톤 MemoryPool에게 40바이트짜리 메모리를 "빌려줘" 요청
4
IndexFromSize(40) — 풀 인덱스 계산
40바이트 → log2(40)=5.3 → ceil=6 → 64바이트 풀 선택 (idx=2)
5
SegmentPool::Rent() — 64바이트 풀에서 빈 세그먼트 꺼냄
스택(free list)에서 하나 pop. 비어있으면 새 chunk 할당
6
return seg._Data — 포인터 반환
Segment 구조체의 데이터 영역 시작 주소를 돌려줌
7
생성자 실행! — 반환된 메모리 위에 TestObject 초기화
멤버 변수 세팅, 가상 함수 테이블(vptr) 설정 등
new TestObject() │ ▼ Object::operator new(40) │ ▼ MemoryPool::Rent(40) │ ├─▶ IndexFromSize(40) → idx=2 (64B pool) │ ▼ SegmentPool[2]::Rent() │ ├─▶ free list에서 pop (or 새 chunk 할당) │ ▼ return seg._Data ← 64바이트 메모리 블록 포인터 │ ▼ TestObject::TestObject() ← 생성자가 이 메모리 위에서 실행
40바이트 요청인데 64바이트를 주는 이유?
메모리 풀은 2의 거듭제곱 크기(16, 32, 64, 128...)로 미리 잘라놓는다. 40바이트는 32바이트 풀에 안 들어가니까, 그 다음 크기인 64바이트 풀에서 빌려준다. 24바이트가 낭비되지만, 관리가 훨씬 단순해지고 빠르다. 이것을 내부 단편화(internal fragmentation)라고 한다.

포인터 산술

SegmentPool이 새 chunk를 할당할 때, 큰 메모리 덩어리를 받아서 일정 간격으로 잘라 각 세그먼트의 시작 주소를 계산한다. 이게 포인터 산술(pointer arithmetic)의 핵심 사용법이다.

C++ SegmentPool::Create() — Chunk 분할 SegmentPool.cpp
byte* chunk = new byte[_BuffSize * _BucketSize]();
_Chunks.push_back(chunk);

for (size_t i = 0; i < _BucketSize; ++i)
{
    FSegment seg;
    seg._Data = chunk + (i * _BuffSize);  // ← 포인터 산술!
    seg._Size = _BuffSize;
    seg._IsPooled = true;

    _Buckets.push(seg);
}

chunk + (i * _BuffSize)가 뭘 의미하는가?

chunk는 byte* (unsigned char*)이므로 1바이트 단위로 이동한다. chunk + i * 64는 "chunk 시작 주소에서 i x 64바이트 뒤의 주소"라는 뜻이다.

계산 공식: 주소 = chunk + i × _BuffSize

i=1: 0x1000 + 1 × 64 = 0x1000 + 0x40 = 0x1040
i=2: 0x1000 + 2 × 64 = 0x1000 + 0x80 = 0x1080
각 세그먼트는 정확히 64바이트 간격으로 떨어져 있다.
🅿️ ANALOGY — 주차장
넓은 공터(chunk)를 빌렸다고 생각해봐.
주차 한 칸의 너비가 64m(=_BuffSize)라면,

0번 칸 = 입구에서 0m 지점
1번 칸 = 입구에서 64m 지점
2번 칸 = 입구에서 128m 지점

i번 칸의 위치 = 입구 + i × 칸 너비
이게 포인터 산술의 전부야!
왜 byte* (unsigned char*) 인가?
이 프로젝트에서는 using byte = unsigned char;로 정의했다. byte는 정확히 1바이트이므로, byte*에 숫자를 더하면 정확히 그 바이트 수만큼 주소가 이동한다. 만약 int*(4바이트)였다면 ptr + 1이 4바이트 이동해버려서 계산이 꼬인다.

2의 거듭제곱 정렬

MemoryPool은 풀을 16, 32, 64, 128, 256, 512 바이트 크기로 만든다. 전부 2의 거듭제곱이다. 임의의 크기 요청이 들어오면 그 크기 이상인 가장 작은 2의 거듭제곱을 찾아야 한다.

C++ IndexFromSize 함수 MemoryPool.cpp
int MemoryPool::IndexFromSize(int size)
{
    int log = static_cast<int>(std::ceil(std::log2(size)));
    int idx = log - _DefaultPow;  // _DefaultPow = 4
    return idx;
}

왜 2의 거듭제곱인가?
  • 분류가 간단 — log2 한 번이면 어떤 풀인지 바로 알 수 있다
  • 풀 수가 적다 — 16~512B까지 커버해도 풀이 6개뿐
  • CPU 캐시 정렬 — 2의 거듭제곱은 캐시 라인에 자연스럽게 정렬됨
  • 비트 연산 최적화 — 나눗셈 대신 시프트(shift)로 처리 가능

_DEBUG new 재정의 문제

Visual Studio의 디버그 모드에서는 메모리 누수를 추적하기 위해 new 키워드를 매크로로 재정의한다. 이게 우리의 operator new와 심각하게 충돌한다.

C++ 디버그 모드 new 매크로 Client.cpp (상단)
#ifdef _DEBUG
#include <crtdbg.h>
#define new new(_NORMAL_BLOCK, __FILE__, __LINE__)
#endif
CRITICAL — 매크로 충돌
이 매크로가 활성화되면 모든 new 키워드가 치환된다.
우리가 정의한 operator new의 선언 자체가 망가진다!

무엇이 깨지는가?

매크로 충돌 — Before / After
BEFORE MACRO (정상)
void* operator new(size_t size)
OK — 정상적인 operator new 선언
⬇ #define new new(_NORMAL_BLOCK, __FILE__, __LINE__) 적용
AFTER MACRO (깨짐!)
void* operator new(_NORMAL_BLOCK, __FILE__, __LINE__)(size_t size)
COMPILE ERROR — new가 매크로로 치환되어 문법이 완전히 망가짐!

#define new ...는 전처리기(preprocessor) 레벨에서 작동한다. 컴파일러가 코드를 보기도 전에, 소스 코드의 모든 new라는 텍스트가 치환된다. operator new라는 함수 선언 안의 new까지 바뀌어 버리는 것이다.

해결 방법

C++ 해결책 — operator new 정의 전에 매크로 해제 Object.h
// 방법 1: operator new를 정의하는 파일에서 매크로 해제
#ifdef _DEBUG
#undef new                // ← 디버그 매크로 해제!
#endif

void* Object::operator new(size_t size)
{
    FSegment seg = MemoryPool::Instance().Rent((int)size);
    Object* obj = (Object*)seg._Data;
    obj->_Segment = seg;
    return seg._Data;
}
C++ 더 깔끔한 방법 — push/pop macro Object.h
#pragma push_macro("new")    // 현재 new 매크로를 스택에 저장
#undef new                    // new 매크로 해제

void* Object::operator new(size_t size)
{
    FSegment seg = MemoryPool::Instance().Rent((int)size);
    Object* obj = (Object*)seg._Data;
    obj->_Segment = seg;
    return seg._Data;
}

void Object::operator delete(void* ptr) noexcept
{
    Object* obj = (Object*)ptr;
    MemoryPool::Instance().Return(obj->_Segment);
}

#pragma pop_macro("new")     // 저장했던 new 매크로 복원
push_macro / pop_macro
#pragma push_macro("new")는 현재 new 매크로의 정의를 스택에 임시 저장한다. #undef new로 해제한 뒤 operator new를 안전하게 정의하고, #pragma pop_macro("new")로 원래 매크로를 복원한다.

이러면 이 파일의 나머지 코드에서는 디버그 매크로가 다시 작동하므로, 메모리 누수 추적도 정상적으로 유지된다. 가장 안전한 방법이다!
🎭 ANALOGY — 변장과 정체
#define new ...는 "new"라는 이름에 변장을 씌우는 것이다.
operator new를 정의할 때는 진짜 new가 필요한데, 변장한 가짜가 나타나서 혼란이 생긴다.

push_macro = "변장 도구를 잠깐 서랍에 넣어"
undef = "변장 벗어"
pop_macro = "서랍에서 변장 도구 다시 꺼내"

정의 구간에서만 맨얼굴(진짜 new)을 보고, 나머지는 다시 변장(디버그 매크로) 유지!

전체 요약

SUMMARY
operator new/delete가 하는 일
  1. operator new를 Object 클래스에 정의하면, 모든 자식 클래스의 new를 가로챌 수 있다
  2. 가로챈 new에서 MemoryPool::Rent()를 호출하면, OS 대신 풀에서 메모리를 빌려온다
  3. operator delete에서 MemoryPool::Return()을 호출하면, 메모리를 OS에 돌려주지 않고 풀에 반납한다
  4. 풀은 2의 거듭제곱 크기로 관리되며, IndexFromSize()로 적절한 풀을 고른다
  5. 포인터 산술로 chunk 내부를 동일 크기의 세그먼트로 분할한다
  6. _DEBUG 모드의 new 매크로와 충돌하므로 push/pop_macro로 우회한다
