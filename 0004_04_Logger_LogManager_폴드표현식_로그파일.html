Logger + LogManager
폴드 표현식 + 로그 파일

C++17 폴드 표현식이 처음 등장. 로그 메시지 조합부터 파일 로테이션까지, 로그 시스템 전체를 해부.

로그 파이프라인

로그가 흐르는 경로
1
LogManager::Debug("HP", 100, "/", 200) — 사용 코드에서 호출
2
Logger::Logging() — 레벨 필터링 (Level > level이면 무시)
3
Logger::WriteLog() — 시간 + 레벨 + 폴드 표현식으로 데이터 결합

로그 레벨

C++ 레벨 정의 Core/Logger.h
enum class eLogLevel
{
    Debug,      // 개발 중 디버깅용
    Info,       // 일반 정보
    Warning,    // 경고
    Error,      // 에러 (기능 실패)
    Fatal,      // 치명적 (프로그램 종료급)
    END
};

#define DEBUG_LEVEL "[DEBUG]"
#define INFO_LEVEL "[INFO]"
#define WARNING_LEVEL "[WARNING]"
#define ERROR_LEVEL "[ERROR]"
#define FATAL_LEVEL "[FATAL]"

Logger::Logging — 레벨 필터링 + WriteLog 호출

C++ Logging 함수 Core/Logger.h
template<typename T, typename ...Args>
void Logging(eLogLevel level, const T& data, const Args& ...args)
{
    if (Level > level)    // ← 레벨 필터링!
        return;

    switch (level)
    {
    case eLogLevel::Debug:
        WriteLog(DEBUG_LEVEL, data, args...);
        break;
    case eLogLevel::Info:
        WriteLog(INFO_LEVEL, data, args...);
        break;
    // ... Warning, Error, Fatal 동일 패턴
    }
}

WriteLog — 시간 + 레벨 + 데이터 조합

C++ WriteLog (시간+레벨 조합) Core/Logger.h
template<typename T, typename ...Args>
void WriteLog(const char* logLevelString, const T& data, const Args& ...args)
{
    // 로컬시간을 얻어오는 객체
    SYSTEMTIME sysTime;
    GetLocalTime(&sysTime);

    char timeText[128] = {};
    sprintf_s(timeText, "[%d.%d.%d.%d.%d.%d]",
        sysTime.wYear, sysTime.wMonth, sysTime.wDay,
        sysTime.wHour, sysTime.wMinute, sysTime.wSecond);

    std::ostringstream stream;
    stream << timeText;        // [2026.3.24.14.30.5]
    stream << logLevelString;  // [DEBUG]

    // 매개변수로 들어온 로그 정보들을 stream에 넣어줌
    WriteLog(stream, data, args...);

    Logs.push(stream.str());   // 완성된 문자열을 큐에 push
}

C++17 폴드 표현식 — 핵심

C++ WriteLog (데이터 결합) — 폴드 표현식 Core/Logger.h
template<typename T, typename ...Args>
void WriteLog(std::ostringstream& os, const T& data, const Args& ...args)
{
    os << data;

    // C++17 폴드표현식
    ((os << "," << args), ...);
}

왜 ((...), ...) 이중 괄호인가?

C++ 폴드 표현식 구문 분석
// 폴드 표현식 문법:
// ( expression , ... )
//   ↑ 이 expression 전체가 하나의 "패턴"

(( os << "," << args ), ...)
 ^                   ^
 내부 괄호: "os << "," << args" 전체가
           하나의 표현식이라는 뜻
           (쉼표 연산자와 구분하기 위해)

쉼표 연산자(,)의 역할

C++에서 ,는 연산자로 쓸 수 있어: 왼쪽 실행 → 버리고 → 오른쪽 결과 반환.

C++ 쉼표 연산자 예시
int x = (1, 2, 3);  // x = 3 (앞의 1, 2는 실행 후 버려짐)

폴드 표현식에서 ,는 각 args에 대해 순서대로 실행하라는 의미:

C++ 전개 결과
// ((os << "," << args), ...) 가 전개되면:
((os << "," << 100), (os << "," << "/"), (os << "," << 200));
//  실행1                실행2                  실행3

Tick — 큐 → 파일

C++ Tick() Common/LogManager.cpp
#define MAX_LINES 1000

void LogManager::Tick(bool bPrint)
{
    while (!_logger.Logs.empty())
    {
        std::string msg = _logger.Logs.front();
        _logger.Logs.pop();

        if (!_file.is_open())
            continue;

        _file << msg << "\n";
        _lineCount++;

        if (_lineCount > MAX_LINES)
        {
            _lineCount = 0;
            _fileIndex++;
            OpenNewFile();
        }
    }
}

OpenNewFile

C++ 새 로그 파일 열기 Common/LogManager.cpp
void LogManager::OpenNewFile()
{
    auto logPath = DirectoryManager::Instance().GetPath("Logs");
    if (!logPath)     // std::optional — 없으면 return
        return;

    SYSTEMTIME sysTime;
    GetLocalTime(&sysTime);

    char fileName[128] = {};
    sprintf_s(fileName, "%d_%d_%d_%d_%d_%d.txt",
        sysTime.wYear, sysTime.wMonth, sysTime.wDay,
        sysTime.wHour, sysTime.wMinute, sysTime.wSecond);

    std::filesystem::path filePath = *logPath / fileName;  // optional 역참조 + 경로 결합

    if (_file.is_open())
        _file.close();

    _file.open(filePath, std::ios::out | std::ios::app);
}
여기서 Day04의 다른 개념들이 합류
DirectoryManager::GetPath() → std::optional 반환
*logPath / fileName → std::filesystem::path 경로 결합
std::ios::out | std::ios::app → 쓰기 모드 + 이어쓰기 (append)

Init / Destroy

C++ Common/LogManager.cpp
void LogManager::Init()
{
    _logger.Init(eLogLevel::Debug);  // 디버그 레벨부터 전부 기록
    OpenNewFile();
}

void LogManager::Destroy()
{
    Tick(false);  // 남은 큐를 전부 파일에 쓰기
}
Destroy에서 Tick(false) 호출하는 이유
프로그램 종료 전에 큐에 남은 로그를 빠짐없이 저장. 안 하면 마지막 몇 줄이 유실될 수 있어.

왜 바로 파일에 안 쓰고 큐를 거치나?

바로 파일에 쓰면
로그 쓸 때마다 파일 I/O 발생
게임 로직 중간에 디스크 접근 → 프레임 드랍
멀티스레드 확장 불가
큐를 거치면
로그 쓰는 코드(push)와 파일 저장(Tick)이 분리
Tick에서 한꺼번에 쓰기 → I/O 최소화
나중에 별도 IO 스레드로 확장 가능
