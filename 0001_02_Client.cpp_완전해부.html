Client.cpp
완전 해부

이 코드가 왜 무서워 보이는 걸까? — Win32 API가 1990년대에 설계된 C 스타일 API라서 그래. 타입 이름이 대문자 덩어리고, 매크로가 난무하고, 함수 이름이 길어. 하지만 실제로 하는 일은 딱 하나 — 빈 창 하나 띄우고 이벤트 기다리기.

전체 흐름

프로그램 시작 │ ▼ wWinMain() ← main() 대신 이걸 씀 │ ├─▶ MyRegisterClass() ← "이런 창 만들 거야" (설계도 등록) │ ├─▶ InitInstance() ← "진짜 만들어" (CreateWindowW) │ └─▶ 무한루프 (게임루프) ├─ 이벤트 있음 → WndProc()에서 처리 └─ 이벤트 없음 → 게임 로직 (아직 비어있음)

Part 1 — 헤더와 전역 변수

파일 상단의 #include와 전역 변수들.

C++ 헤더 & 디버그 매크로 Client.cpp:1-17
#include "framework.h"     // Windows API 기본 헤더 모음
#include "Client.h"        // Resource.h 포함 (메뉴, 아이콘 ID)

#ifdef _DEBUG
#include <crtdbg.h>
#define new new(_NORMAL_BLOCK, __FILE__, __LINE)
#endif                        // ↑ 디버그 모드에서 메모리 누수 추적. 지금은 무시.

HINSTANCE hInst;               // "이 프로그램 자체"의 ID (핸들)
WCHAR     szTitle[100];       // 창 제목 문자열
WCHAR     szWindowClass[100]; // 창 클래스 이름 (OS 구분용)
HINSTANCE가 뭔데?
Windows에서 실행 중인 프로그램(프로세스)을 구분하는 ID.
메모장을 2개 열면 각각 다른 HINSTANCE를 가진다.
H = Handle (핸들), INSTANCE = 실행된 프로그램 하나.

Part 2 — wWinMain (진입점)

이건 그냥 main() 함수! Windows GUI 프로그램에서는 main() 대신 wWinMain()을 쓰는 것뿐.

C++ 진입점 함수 시그니처 Client.cpp:26-29
int APIENTRY wWinMain(
    _In_     HINSTANCE hInstance,      // ✅ 이 프로그램의 핸들 (계속 씀)
    _In_opt_ HINSTANCE hPrevInstance,  // ❌ 항상 NULL (16비트 잔재)
    _In_     LPWSTR    lpCmdLine,      // ❌ 명령줄 인자 (안 씀)
    _In_     int       nCmdShow        // ✅ 창 표시 방법
)
SAL ANNOTATION
_In_, _In_opt_ 같은 건 SAL 어노테이션이라고 한다. 컴파일러에게 "이 매개변수는 입력용이야", "이건 선택적이야" 라고 힌트를 주는 건데, 실행에는 아무 영향 없다. 장식이라고 생각하면 됨.

안에서 하는 일

Part 3 — MyRegisterClass (설계도 등록)

WNDCLASSEXW 구조체에 창의 옵션을 채워서 OS에 등록하는 함수. 비유하면 건축 주문서 작성.

C++ 창 설계도 Client.cpp:111-131
WNDCLASSEXW wcex;

wcex.cbSize        = sizeof(WNDCLASSEX);       // 구조체 크기
wcex.style         = CS_HREDRAW | CS_VREDRAW; // 크기 바뀌면 다시 그려
wcex.lpfnWndProc   = WndProc;                 // ★ 이벤트 처리 함수 (콜백)
wcex.hIcon         = LoadIcon(...);             // 아이콘
wcex.hCursor       = LoadCursor(...);           // 커서 (기본 화살표)
wcex.hbrBackground = (HBRUSH)(COLOR_WINDOW+1); // 배경색 (흰색)
wcex.lpszClassName = szWindowClass;             // 클래스 이름

RegisterClassExW(&wcex);  // OS에 등록!
CRITICAL — 가장 중요한 줄
wcex.lpfnWndProc = WndProc;
"이 창에 무슨 일이 생기면 WndProc 함수를 불러줘"라고 OS에 알려주는 것. 함수 포인터(콜백)라는 개념인데, 이후 DX11 전체에서 아주 중요하게 쓰인다.

Part 4 — InitInstance (창 생성)

C++ CreateWindowW 해부 Client.cpp:158-159
HWND hWnd = CreateWindowW(
    szWindowClass,          // 아까 등록한 설계도 이름
    szTitle,                // 창 제목
    WS_OVERLAPPEDWINDOW,    // 일반 창 스타일 (제목바+테두리+최소화/최대화)
    CW_USEDEFAULT, 0,       // 위치: OS가 알아서
    CW_USEDEFAULT, 0,       // 크기: OS가 알아서
    nullptr,                // 부모 창: 없음
    nullptr,                // 메뉴: 설계도에 있는 거 쓸게
    hInstance,              // 이 프로그램의 핸들
    nullptr                 // 추가 데이터: 없음
);

ShowWindow(hWnd, nCmdShow);  // 창을 화면에 보여줘
UpdateWindow(hWnd);          // 한번 그려줘 (WM_PAINT 발생)
🏗️ ANALOGY — 커널 오브젝트와 핸들
Windows OS가 관리하는 것들(창, 파일, 프로세스 등)을 커널 오브젝트라고 해.
직접 건드릴 수 없고, OS가 발급해준 ID(핸들)로만 조작할 수 있어.

비유: 은행 계좌를 직접 만질 수 없고, 계좌번호(핸들)로만 입출금하는 것과 같아.

왜 HWND, HPEN, HBRUSH처럼 타입을 나눴냐면?
숫자 자체론 구분이 안 되니까 타입으로 안전장치를 건 거야.
(HWND에 HPEN을 실수로 넣는 걸 방지)

Part 5 — 게임 루프

C++ PeekMessage 루프 Client.cpp:75-98
while (true)
{
    if (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE))
    {
        if (msg.message == WM_QUIT)
            break;

        if (!TranslateAccelerator(msg.hwnd, hAccelTable, &msg))
        {
            TranslateMessage(&msg);  // 키보드 → 문자 변환
            DispatchMessage(&msg);   // WndProc에 전달 ← 핵심!
        }
    }
    else
    {
        // 윈도우 데드타임 → 게임 로직!
        // Logic();
    }
}
🏪 ANALOGY — 편의점 알바
너 = 편의점 알바생 (프로그램)
손님 = 윈도우 메시지 (이벤트)

while (가게 문 열려있는 동안)
  if (손님 왔나 확인 — PeekMessage)
    if (사장님이 "닫아" 했으면 — WM_QUIT) → 퇴근
    손님 응대 → DispatchMessage → WndProc 호출
  else
    손님 없으니까 진열대 정리 — 게임 로직!

PeekMessage = 힐끗 보고 없으면 다른 일 함
GetMessage = 손님 올 때까지 멍하니 기다림
TranslateMessage / DispatchMessage
TranslateMessage → 키보드 입력을 문자 메시지로 변환 (WM_KEYDOWN → WM_CHAR)
DispatchMessage → 메시지를 WndProc 함수에 전달 (이게 핵심!)

Part 6 — WndProc (이벤트 처리기)

이 함수는 네가 부르는 게 아니라 OS가 부르는 함수(콜백).

C++ WndProc 구조 Client.cpp:189-235
LRESULT CALLBACK WndProc(HWND hWnd, UINT message,
                          WPARAM wParam, LPARAM lParam)
{
    switch (message)
    {
    case WM_COMMAND:   // 메뉴 클릭
        // IDM_ABOUT → 정보 대화상자
        // IDM_EXIT  → 창 파괴
        break;

    case WM_PAINT:     // 창을 (다시) 그릴 때
        // BeginPaint / EndPaint 사이에 그리기 (아직 비어있음)
        break;

    case WM_DESTROY:   // 창이 파괴될 때
        PostQuitMessage(0); // → WM_QUIT → 게임 루프 break
        break;

    default:
        return DefWindowProc(hWnd, message, wParam, lParam);
    }
}
WM_DESTROY vs WM_QUIT
WM_DESTROY = "창이 파괴되고 있다" (이벤트)
WM_QUIT = "프로그램을 종료해라" (명령)

X 버튼 클릭 → WM_DESTROY 발생 → PostQuitMessage() 호출 → WM_QUIT이 메시지 큐에 들어감 → 게임 루프의 break에 도달

Win32 타입 사전

Win32에서 제일 무서운 게 타입 이름. 여기 정리해둘 테니 참고해.

NAMING RULE
H로 시작하면 핸들 / LP로 시작하면 포인터 / W가 붙으면 유니코드 버전
Win32 타입 실제 정체 의미
HINSTANCE void* 프로그램 핸들
HWND void* 창(Window) 핸들
UINT unsigned int 부호 없는 정수
WPARAM unsigned __int64 메시지 부가 정보 1
LPARAM __int64 메시지 부가 정보 2
LRESULT __int64 함수 반환값
BOOL int 0=FALSE, 그 외=TRUE
WCHAR wchar_t 유니코드 문자 (2바이트)
LPWSTR wchar_t* 유니코드 문자열 포인터
ATOM unsigned short RegisterClass 반환값
HACCEL void* 단축키 테이블 핸들
HDC void* 그리기 도구(Device Context) 핸들
CALLBACK __stdcall 호출 규약 (OS가 부르는 함수 표시)
APIENTRY __stdcall CALLBACK이랑 같은 것
