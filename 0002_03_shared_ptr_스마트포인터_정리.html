shared_ptr / weak_ptr
enable_shared_from_this

스마트 포인터 — 메모리 누수와 댕글링 포인터를 자동으로 방지하는 C++ 무기

왜 스마트 포인터?

C++에서 new로 메모리를 할당하면, 반드시 delete로 해제해야 한다. 이걸 잊으면? 메모리 누수(memory leak). 두 번 하면? 크래시(double free). 포인터가 가리키던 메모리가 이미 해제됐는데 접근하면? 댕글링 포인터(dangling pointer).

게임 엔진처럼 오브젝트가 수백 개 생기고 사라지는 환경에서 이걸 수동으로 관리하는 건 지뢰밭 위를 뛰어다니는 것과 같다. 그래서 C++11부터 스마트 포인터가 표준에 들어왔다.

생 포인터(raw pointer)의 문제

C++ 메모리 누수 예시
void Dangerous()
{
    TestObject* obj = new TestObject();  // 힙에 할당

    // ... 여기서 예외가 터지거나 return을 하면?
    if (someCondition)
        return;  // delete를 안 하고 빠져나감!

    delete obj;  // 여기까지 안 올 수도 있다
}
생 포인터 — 메모리 누수 시각화

Step 1 — 함수 진입: 포인터가 힙 메모리를 가리킴

obj -->
-->
TestObject

Step 2 — 함수 중간에 return: 포인터(스택)는 사라지고, 메모리(힙)는 남음

obj X
-->
LEAKED!

delete를 안 했으니 메모리가 프로그램 종료까지 영원히 점유된다. 이게 메모리 누수.

생 포인터의 3대 재앙
1. 메모리 누수 — delete 안 하면 메모리가 계속 쌓인다
2. 댕글링 포인터 — 이미 해제된 메모리를 가리키는 포인터로 접근하면 정의되지 않은 동작
3. Double Free — 같은 메모리를 두 번 delete하면 크래시

shared_ptr

shared_ptr은 참조 카운트(reference count) 기반 스마트 포인터다. 여러 개의 shared_ptr이 하나의 객체를 공유할 수 있고, 마지막 shared_ptr이 소멸될 때 자동으로 객체가 삭제된다.

📚 ANALOGY — 도서관 대출 카드
shared_ptr = 대출 카드
책을 빌리면 대출 카운터 +1, 반납하면 -1.
카운터가 0이 되면 (아무도 안 빌리면) 그 책은 폐기 처분된다.

weak_ptr = 열람실에서 보기만 하는 것
대출 카운터에 포함되지 않는다. 책이 폐기되면 "없음" 표시가 뜬다.
열람실에서 보고 있다고 해서 책이 폐기 안 되는 건 아니다!

기본 사용법

C++ shared_ptr 기본
#include <memory>

class TestObject
{
public:
    TestObject()  { /* 생성 */ }
    ~TestObject() { /* 소멸 — shared_ptr이 자동 호출 */ }
};

void Example()
{
    // make_shared: 객체 + 컨트롤 블록을 한 번에 할당 (효율적!)
    auto obj = make_shared<TestObject>();

    // use_count()로 현재 참조 카운트 확인
    cout << obj.use_count() << endl;  // 1

    {
        auto obj2 = obj;  // 복사 → ref_count: 2
        cout << obj.use_count() << endl;  // 2
    }   // obj2 스코프 벗어남 → ref_count: 1

    cout << obj.use_count() << endl;  // 1
}   // obj 소멸 → ref_count: 0 → delete 자동 호출!
make_shared vs new
make_shared<T>()는 객체와 컨트롤 블록(참조 카운트 등)을 한 번의 메모리 할당으로 만든다.
shared_ptr<T>(new T())는 두 번 할당한다 (객체 + 컨트롤 블록 따로).
성능과 예외 안전성 모두 make_shared가 우위. 특별한 이유 없으면 항상 make_shared를 쓰자.

ref_count 동작 시각화

State 1 — auto obj = make_shared<TestObject>();
shared_ptr obj -->
-->
TestObject   ref:1

shared_ptr 하나가 객체를 가리킴. 참조 카운트 = 1.

State 2 — auto obj2 = obj; (복사)
shared_ptr obj -->
-->
TestObject   ref:2
<--
shared_ptr obj2 <--

두 개의 shared_ptr이 같은 객체를 가리킴. 참조 카운트 = 2. 객체는 하나뿐!

State 3 — obj.reset(); (obj 해제)
shared_ptr obj nullptr
X
TestObject   ref:1
<--
shared_ptr obj2 <--

obj가 소유권을 포기함. 아직 obj2가 가리키고 있으니 참조 카운트 = 1. 객체는 살아있다!

State 4 — obj2.reset(); (마지막 참조 해제)
shared_ptr obj nullptr
X
DELETED   ref:0
X
shared_ptr obj2 nullptr

참조 카운트 0 → 소멸자 자동 호출 → 메모리 해제! 아무도 관리 안 해도 누수 없음.

주요 멤버 함수

함수 설명 예시
use_count() 현재 참조 카운트 반환 obj.use_count()
reset() 소유권 포기 (카운트 -1) obj.reset()
get() 생 포인터(raw pointer) 반환 T* raw = obj.get()
unique() 참조 카운트가 1이면 true (C++17 deprecated) obj.unique()
operator bool nullptr이 아니면 true if (obj) { ... }

weak_ptr

weak_ptr은 shared_ptr이 관리하는 객체를 참조 카운트를 올리지 않고 관찰하는 포인터다. 객체의 수명에 영향을 주지 않으면서, 객체가 아직 살아있는지 확인할 수 있다.

왜 필요한가? — 순환 참조 문제

두 객체가 서로를 shared_ptr로 가리키면, 참조 카운트가 절대 0이 되지 않는다. 누구도 먼저 죽을 수 없다. 이것이 순환 참조(circular reference) 문제다.

순환 참조 — 둘 다 절대 해제 안 됨
Object A shared_ptr<B> --> ref:1
--shared_ptr--> <--shared_ptr--
Object B shared_ptr<A> --> ref:1

외부 shared_ptr 해제 후에도: A의 ref_count = 1 (B가 가리킴), B의 ref_count = 1 (A가 가리킴)

CRITICAL — 순환 참조 = 메모리 누수
A가 B를 shared_ptr로, B가 A를 shared_ptr로 가리키면, 외부에서 둘 다 놓아줘도 서로가 서로를 잡고 있어서 ref_count가 0이 안 된다.
결과: 소멸자 안 불림, 메모리 영원히 점유. 이건 스마트 포인터를 써도 막을 수 없는 함정!

해결: weak_ptr로 한쪽을 약한 참조로

weak_ptr로 해결 — 한쪽이 약한 참조
Object A shared_ptr<B> --> ref:1
--shared_ptr--> <--weak_ptr----
Object B weak_ptr<A> (카운트 안 올림!) ref:1

B의 weak_ptr은 A의 ref_count에 포함 안 됨 → A 삭제 가능 → B의 weak_ptr은 expired 상태가 됨

weak_ptr 사용법 — lock() 패턴

weak_ptr은 직접 역참조(*, ->)를 할 수 없다. 반드시 lock()을 통해 shared_ptr로 변환한 뒤 사용해야 한다. 이미 객체가 소멸됐으면 lock()은 빈 shared_ptr을 반환한다.

C++ weak_ptr lock() 패턴
class Node
{
public:
    shared_ptr<Node> child;     // 자식은 강한 참조
    weak_ptr<Node>   parent;    // 부모는 약한 참조 (순환 방지!)

    void DoSomethingWithParent()
    {
        // lock()으로 shared_ptr 획득 시도
        auto sp = parent.lock();

        if (sp)
        {
            // 부모가 아직 살아있음 → 안전하게 사용
            sp->SomeMethod();
        }
        else
        {
            // 부모가 이미 소멸됨 → expired 상태
            // 댕글링 포인터 접근 없이 안전하게 처리!
        }
    }
};
weak_ptr 핵심 정리
expired() — 객체가 이미 소멸됐으면 true 반환
lock() — 아직 살아있으면 shared_ptr 반환, 소멸됐으면 빈 shared_ptr
use_count() — 강한 참조(shared_ptr) 개수 반환 (weak 포함 아님!)
C++ 순환 참조 해결 전체 코드
class A;
class B;

class A
{
public:
    shared_ptr<B> ptrB;  // 강한 참조
    ~A() { cout << "A 소멸" << endl; }
};

class B
{
public:
    weak_ptr<A> ptrA;  // ★ 약한 참조! 순환 깨뜨림
    ~B() { cout << "B 소멸" << endl; }
};

void Test()
{
    auto a = make_shared<A>();
    auto b = make_shared<B>();

    a->ptrB = b;    // A → B (shared_ptr, ref_count 증가)
    b->ptrA = a;    // B → A (weak_ptr, ref_count 안 증가!)
}
// 함수 끝: a 소멸 → A ref 0 → A 삭제 → B ref 0 → B 삭제
// 출력: "A 소멸" → "B 소멸" ✅

enable_shared_from_this

멤버 함수 안에서 자기 자신(this)의 shared_ptr을 만들고 싶을 때가 있다. 예를 들어 이벤트 시스템에 자기 자신을 등록하거나, 다른 객체에게 자기를 넘겨줄 때.

잘못된 방법 (CRASH!)

C++ 절대 하면 안 되는 것!
class Object
{
public:
    shared_ptr<Object> GetSelf()
    {
        // ❌ 절대 이러면 안 됨!
        return shared_ptr<Object>(this);
    }
};
잘못된 방법 — shared_ptr(this) → DOUBLE DELETE!

문제: 서로 다른 컨트롤 블록(ref_count)이 두 개 생김!

original -->
-->
Object 컨트롤 블록 A: ref:1
<--
shared_ptr(this) <--
↑ 별도의 컨트롤 블록 B: ref:1 (위의 것과 다름!)

original과 shared_ptr(this)가 각각 ref_count를 독립적으로 관리 → 둘 다 delete 시도 → CRASH!

올바른 방법: enable_shared_from_this 상속

C++ enable_shared_from_this 올바른 사용
class Object : public std::enable_shared_from_this<Object>
{
public:
    // ✅ 기존 컨트롤 블록을 공유! ref_count가 올라감
    // shared_from_this()로 안전하게 자기 자신의 shared_ptr을 얻음
};
올바른 방법 — shared_from_this() → 같은 컨트롤 블록!

핵심: 같은 컨트롤 블록을 공유해서 ref_count만 올라감

original -->
-->
Object 컨트롤 블록 (하나!): ref:2 ref:2
<--
shared_from_this() <--

두 shared_ptr이 동일한 컨트롤 블록을 공유 → ref_count 정확 → 안전한 소멸!

프로젝트에서 사용하는 This<T>() 패턴

shared_from_this()는 리턴 타입이 shared_ptr<Base>다. 파생 클래스에서 shared_ptr<Derived>가 필요할 때는 다운캐스트가 필요하다. 이걸 매번 쓰기 번거로우니 템플릿 헬퍼를 만들어둔다.

C++ This<T>() 템플릿 헬퍼 — 실제 코드 Object.h (기본 버전)
class Object : public std::enable_shared_from_this<Object>
{
public:
    template<typename T>
    std::shared_ptr<T> This()
    {
        // shared_ptr의 this
        return std::dynamic_pointer_cast<T>(shared_from_this());
    }

    virtual void Init() = 0;
    virtual void Destroy() = 0;
};

// 사용 예: TestObject 안에서
auto self = This<TestObject>();  // shared_ptr<TestObject>를 안전하게 얻음
주의 — shared_from_this() 사용 조건
shared_from_this()는 이미 shared_ptr로 관리되고 있는 객체에서만 호출 가능하다.
스택에 만든 객체나 new로 만들고 shared_ptr에 넣지 않은 객체에서 호출하면 예외 발생!

C++ 이렇게 하면 크래시
Object* obj = new TestObject();     // ❌ shared_ptr로 관리 안 됨
obj->shared_from_this();              // 💥 bad_weak_ptr 예외!

// 반드시 shared_ptr로 생성된 객체에서만:
auto obj2 = std::make_shared<TestObject>();
obj2->This<TestObject>();             // ✅ OK

using 타입 앨리어스

shared_ptr<SomeVeryLongClassName>을 매번 쓰는 건 손가락이 아프다. 프로젝트에서는 using으로 타입 별칭을 만들어 간결하게 쓴다.

C++ 실제 코드 — 조건부 타입 앨리어스 Info.h
#ifdef _USE_MEMORY_POOL
using TestObjectPtr = class TestObject*;                     // 생 포인터
using WeakTestObject = class TestObject*;                    // 생 포인터
#elif _USE_OBJECT_POOL
using TestObjectPtr = class TestObject*;                     // 생 포인터
using WeakTestObject = class TestObject*;                    // 생 포인터
#else
using TestObjectPtr = std::shared_ptr<class TestObject>;   // 스마트 포인터
using WeakTestObject = std::weak_ptr<class TestObject>;  // 약한 참조
#endif

class TestObject는 전방 선언이야 — "이런 클래스가 어딘가에 있을 거야"라고 알려주는 것. 포인터나 shared_ptr만 쓸 때는 전방 선언으로 충분해. 헤더 순환 의존을 방지할 수 있어.

전방 선언(Forward Declaration)이란?
class Texture;만 적고 #include "Texture.h"는 안 하는 것.
포인터(shared_ptr 포함)만 사용할 때는 전방 선언으로 충분하다. 클래스의 멤버에 접근하지 않고, 크기도 알 필요 없으니까.

장점: 헤더 간 의존성이 줄어들어 컴파일 시간이 빨라진다. 대규모 프로젝트에서 이건 정말 중요하다.
C++ using 활용 예시
// Before (길고 반복적)
shared_ptr<Texture> LoadTexture(const wstring& path);
shared_ptr<Material> CreateMaterial(shared_ptr<Texture> tex);

// After (깔끔!)
TextureRef LoadTexture(const wstring& path);
MaterialRef CreateMaterial(TextureRef tex);

3가지 스마트 포인터 비교

C++11 표준에는 세 가지 스마트 포인터가 있다. 각각의 용도와 특성을 비교해보자.

unique_ptr

독점 소유
오직 하나만 객체를 소유
복사 불가, 이동만 가능
오버헤드 없음 (생 포인터와 동일)
unique_ptr<T>

shared_ptr

공유 소유
여러 포인터가 하나의 객체를 공유
참조 카운트로 수명 관리
약간의 오버헤드 (컨트롤 블록)
shared_ptr<T>

weak_ptr

관찰만
소유하지 않고 관찰만
ref_count에 포함 안 됨
순환 참조 해결용
weak_ptr<T>

항목 unique_ptr shared_ptr weak_ptr
소유권 독점 (1개만) 공유 (N개) 없음 (관찰만)
복사 가능 X (이동만) O O
ref_count 영향 — +1 변화 없음
자동 해제 스코프 끝 ref_count 0일 때 해제 안 함
메모리 오버헤드 0 (포인터 크기) 컨트롤 블록 (16~40B) shared_ptr과 공유
주 사용처 팩토리, 단독 소유 리소스 공유 리소스, 게임 오브젝트 순환 참조 방지, 캐시
DX11 프로젝트에서의 선택 기준
shared_ptr을 기본으로 사용하되, 소유권이 명확히 1곳인 경우는 unique_ptr이 더 가볍다.
부모-자식 관계에서 자식 → 부모 방향은 반드시 weak_ptr을 쓰자.
리소스 매니저(텍스처, 메시 캐시 등)에서도 weak_ptr로 캐시하면, 더 이상 쓰이지 않는 리소스가 자동으로 해제된다.
