VS 프로젝트 설정 해부
slnx / vcxproj / filters

강사님이 폴더 구조를 만들면서 프로젝트 설정도 상당히 건드셨다. 어떤 설정을 왜 했는지, 기본값과 뭐가 다른지.

파일 3개의 역할

sln
Game2D.slnx
솔루션 — 빌드 구성 + 프로젝트 목록
vcx
Client.vcxproj
프로젝트 — 컴파일 옵션, 매크로, 파일 목록
flt
Client.vcxproj.filters
필터 — VS 탐색기 가상 폴더 구조

1. Game2D.slnx — 솔루션 파일

XML 솔루션 빌드 구성 Game2D.slnx
<Solution>
  <Configurations>
    <BuildType Name="Debug" />
    <BuildType Name="Debug_MP" />
    <BuildType Name="Debug_OP" />
    <BuildType Name="Release" />
    <Platform Name="x64" />
    <Platform Name="x86" />
  </Configurations>
  <Project Path="Client/Client.vcxproj" Id="fae35489-..." />
</Solution>

VS에서 프로젝트를 새로 만들면 Debug/Release 2개만 있어. 강사님이 2개를 추가한 거:

빌드 구성기본 제공용도
DebugO기본 디버그 (shared_ptr 전략)
ReleaseO최적화 빌드
Debug_MP추가MemoryPool 전략 (_USE_MEMORY_POOL)
Debug_OP추가ObjectPool 전략 (_USE_OBJECT_POOL)

플랫폼: x64(64비트) + x86(32비트). 조합하면 총 4 구성 x 2 플랫폼 = 8가지 빌드가 가능.

slnx vs sln
.slnx는 VS 2022 17.10+에서 도입된 XML 기반 솔루션 포맷. 기존 .sln은 독자적인 텍스트 포맷이라 읽기 어려웠는데, XML로 바뀌면서 깔끔해짐.

2. Client.vcxproj — 핵심 빌드 설정

2-1. 전처리기 매크로

이게 3가지 메모리 전략을 전환하는 핵심 메커니즘:

XML 빌드 구성별 전처리기 정의 Client.vcxproj
<!-- Debug (x64) — 기본: shared_ptr -->
<PreprocessorDefinitions>_DEBUG;_WINDOWS;%(PreprocessorDefinitions)</PreprocessorDefinitions>

<!-- Debug_OP (x64) — ObjectPool -->
<PreprocessorDefinitions>_DEBUG;_WINDOWS;_USE_OBJECT_POOL;%(PreprocessorDefinitions)</PreprocessorDefinitions>

<!-- Debug_MP (x64) — MemoryPool -->
<PreprocessorDefinitions>_DEBUG;_WINDOWS;_USE_MEMORY_POOL;%(PreprocessorDefinitions)</PreprocessorDefinitions>

<!-- Release (x64) -->
<PreprocessorDefinitions>NDEBUG;_WINDOWS;%(PreprocessorDefinitions)</PreprocessorDefinitions>
빌드 구성 → 매크로 → 코드 분기 흐름
1
VS에서 Debug_OP | x64 선택
2
vcxproj가 _USE_OBJECT_POOL 매크로를 컴파일러에 전달
3
Defines.h의 #elif _USE_OBJECT_POOL 분기 활성화
4
New<T>() → ObjectPool::Rent<T>() 호출
빌드 구성정의되는 매크로코드에서의 효과
Debug_DEBUG, _WINDOWS#else → shared_ptr
Debug_OP_DEBUG, _WINDOWS, _USE_OBJECT_POOLObjectPool 활성화
Debug_MP_DEBUG, _WINDOWS, _USE_MEMORY_POOLMemoryPool 활성화
ReleaseNDEBUG, _WINDOWSshared_ptr + 최적화

%(PreprocessorDefinitions) — 이전에 정의된 매크로를 상속하겠다는 뜻. 안 넣으면 부모 설정이 날아감.

Win32(x86) 빌드에는 전략 매크로가 없다
x64에서만 _USE_OBJECT_POOL / _USE_MEMORY_POOL이 정의됨. Win32 빌드에서는 Debug_OP/Debug_MP를 선택해도 shared_ptr 전략으로 빌드됨. 요즘 컴퓨터는 거의 다 64비트라 x86 설정은 안 건드신 것으로 보임.

2-2. C++ 언어 표준

XML C++20 설정 Client.vcxproj
<LanguageStandard>stdcpp20</LanguageStandard>

C++20 사용. 기본값은 stdcpp14인데 올린 거. C++20 기능 중 이 프로젝트에서 쓰일 수 있는 것들:

  • std::format (문자열 포맷)
  • Concepts (템플릿 제약)
  • Ranges
  • std::span

2-3. 추가 포함 디렉토리

XML AdditionalIncludeDirectories Client.vcxproj
<AdditionalIncludeDirectories>$(SolutionDir)Client\;%(AdditionalIncludeDirectories)</AdditionalIncludeDirectories>

이게 폴더 구조 리팩토링의 핵심 설정. 이 줄이 없으면:

C++ Include 경로 문제
#include "Core/ObjectPool.h"      // ← Client.cpp에서 이렇게 써야 함
// ObjectPool.h 안에서 다시:
#include "Common/Singleton.h"     // ← Core/ 기준 상대 경로라 안 찾아짐!
📂 ANALOGY — 작업 디렉토리
$(SolutionDir)Client\를 포함 경로에 추가하면, Client 폴더가 루트처럼 작동해서 어디서든 "Common/Singleton.h", "Core/Object.h" 같은 경로로 include 가능.

마치 터미널에서 cd Client 하고 나서 모든 파일을 찾는 것처럼.

2-4. 출력 디렉토리

XML OutDir / IntDir Client.vcxproj
<OutDir>$(SolutionDir)Bin/</OutDir>
<IntDir>$(SolutionDir)Bin\BinObj/</IntDir>
설정기본값변경된 값의미
OutDir$(SolutionDir)$(Configuration)\$(SolutionDir)Bin/실행 파일(.exe) 출력 위치
IntDir$(Configuration)\$(SolutionDir)Bin\BinObj/중간 파일(.obj) 출력 위치

기본값이면 Debug/, Release/ 폴더가 여러 개 생기는데, 전부 Bin/으로 통합한 거. 깔끔하지만, 빌드 구성별 출력이 섞일 수 있는 점은 주의.

2-5. 공통 설정 요약

설정값의미
ConfigurationTypeApplication실행 파일(.exe) 생성
PlatformToolsetv145VS 2022+ 컴파일러
CharacterSetUnicodeWCHAR(2바이트) 문자 사용
WarningLevelLevel3경고 수준 3 (기본적인 경고 표시)
SDLChecktrue보안 개발 수명 주기 검사
ConformanceModetrueC++ 표준 준수 모드
SubSystemWindowsWin32 GUI 앱 (콘솔 아님)
GenerateDebugInformationtrue디버그 정보(.pdb) 생성

2-6. Release에만 있는 최적화 옵션

XML Release 최적화 옵션 Client.vcxproj
<FunctionLevelLinking>true</FunctionLevelLinking>
<IntrinsicFunctions>true</IntrinsicFunctions>
<WholeProgramOptimization>true</WholeProgramOptimization>
옵션의미
FunctionLevelLinking사용하지 않는 함수를 링커가 제거 가능
IntrinsicFunctionsmemcpy 등을 CPU 명령어로 직접 대체 (빠름)
WholeProgramOptimization전체 프로그램을 보고 최적화 (Link-Time Code Generation)
왜 Debug에는 없나?
최적화가 켜지면 컴파일러가 코드를 재배치/합치기 때문에, 디버거에서 한 줄씩 따라갈 수 없게 된다. 변수 값도 "최적화됨"으로 표시되어 확인 불가. Debug 빌드는 반드시 최적화 OFF.

3. vcxproj.filters — 솔루션 탐색기

XML 필터(가상 폴더) 정의 Client.vcxproj.filters
<Filter Include="Common">...</Filter>
<Filter Include="Core">...</Filter>
<Filter Include="Core\Memory">...</Filter>
<Filter Include="Object">...</Filter>

이건 VS 솔루션 탐색기에서 보이는 가상 폴더야. 실제 파일 시스템과 별개.

솔루션 탐색기 (가상) vs 실제 파일 시스템
솔루션 탐색기 (가상)
Common/
  Info.h, Singleton.h
Core/
  Defines.h, Object.h/cpp
  Memory/
    MemoryPool, ObjectPool,
    SegmentPool
Object/
  TestObject.h/cpp
실제 파일 시스템
Common/
  Info.h, Singleton.h
Core/
  Defines.h, Object.h/cpp
  MemoryPool, ObjectPool,
  SegmentPool
← Memory/ 폴더 없음!
Object/
  TestObject.h/cpp

차이점: Core/Memory 필터는 솔루션 탐색기에서만 존재. 실제 파일 시스템에서는 Core/ 바로 아래에 있어. 풀 관련 파일을 시각적으로 묶어서 보기 편하게 한 거.
