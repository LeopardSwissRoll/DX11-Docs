핵심 — 3가지 전략 비교

게임에서 new/delete를 날것 그대로 쓰면 성능이 나빠진다. OS에게 매번 메모리를 달라고 하는 건 느리고, 할당/해제를 반복하면 메모리 단편화(fragmentation)가 생긴다. 그래서 미리 메모리를 확보해두고, 필요할 때 빌려 쓰는 전략을 쓴다.

이 프로젝트에서는 3가지 전략을 구현하고, #ifdef로 전환할 수 있게 만들었다.

언제 뭘 쓰나?
MemoryPool — 기본 전략. 게임 엔진 내부 대부분의 객체에 사용.
ObjectPool — 총알, 이펙트처럼 같은 타입의 객체를 대량으로 생성/파괴하는 경우.
shared_ptr — 리소스 공유(텍스처, 메시 등), 소유권이 불명확할 때. 디버그 모드에서 안전장치로 사용.

SegmentPool 구조

SegmentPool은 MemoryPool의 핵심 부품이다. 하나의 SegmentPool은 고정 크기의 메모리 버킷을 관리한다. 예를 들어 "64바이트 전용 풀"이면, 64바이트짜리 메모리 조각을 미리 여러 개 만들어서 큐에 넣어두고, 요청이 오면 하나씩 꺼내준다.

Chunk와 Bucket

SegmentPool은 메모리를 Chunk 단위로 할당한다. 하나의 Chunk는 여러 개의 Bucket으로 나뉜다. Bucket이 실제로 사용자에게 빌려주는 메모리 조각이다.

🎳 ANALOGY — 볼링장
SegmentPool은 볼링장이야.

Chunk = 볼링장에 놓인 볼링공 세트 (한 번에 여러 개 구매)
Bucket = 개별 볼링공 하나
Queue = 볼링공 반환대 (레일 위에 줄 서 있는 공들)

손님이 오면 반환대에서 공 하나를 꺼내주고(Rent), 게임 끝나면 다시 반환대에 올려놓는다(Return).
공이 부족하면? 볼링공 세트를 추가 구매(새 Chunk 할당)한다.

핵심: 공을 파괴하고 다시 만드는 게 아니라, 돌려쓰는 거야. 이게 메모리 풀의 본질이다.

FSegment 구조체

C++ FSegment — 메모리 조각 정보 SegmentPool.h
using byte = unsigned char;  // 1바이트

struct FSegment
{
    byte* _Data = nullptr;  // 실제 메모리 주소
    unsigned int _Size = 0; // 이 세그먼트의 크기
    bool _IsPooled = false; // 풀에서 할당한것인지 여부
};

SegmentPool 핵심 코드

C++ SegmentPool 클래스 구조 SegmentPool.h
class SegmentPool
{
    friend class MemoryPool;

private:
    SegmentPool(int buffSize, int bucketSize);
    ~SegmentPool();

private:
    int _BuffSize = 0;             // 버킷 하나의 크기(바이트)
    int _BucketSize = 0;           // 버킷의 개수

    std::vector<byte*> _Chunks;   // 실제 할당한 메모리
    std::queue<FSegment> _Buckets; // 대여 가능한 버킷 목록

private:
    void Create();                  // 새 청크를 할당하고 버킷으로 쪼개기

public:
    FSegment Rent();               // 버킷 하나를 대여
    void Return(FSegment& seg);    // 사용한 버킷을 반납
};
C++ Create / Rent / Return 구현 SegmentPool.cpp
void SegmentPool::Create()
{
    // _BuffSize * _BucketSize 만큼의 크기를 동적할당
    byte* chunk = new byte[_BuffSize * _BucketSize]();
    _Chunks.push_back(chunk);

    // 할당한 청크를 버킷으로 쪼개서 큐에 넣는다
    for (size_t i = 0; i < _BucketSize; ++i)
    {
        FSegment seg;
        seg._Data = chunk + (i * _BuffSize);
        seg._Size = _BuffSize;
        seg._IsPooled = true;

        _Buckets.push(seg);
    }
}

FSegment SegmentPool::Rent()
{
    // 버킷이 있다면 큐에서 하나 빌려온다
    if (!_Buckets.empty())
    {
        FSegment seg = _Buckets.front();
        _Buckets.pop();
        return seg;
    }

    // 버킷이 충분하지 않으므로 직접 동적할당하여 제공
    FSegment seg;
    seg._Data = new byte[_BuffSize]();
    seg._IsPooled = false;
    seg._Size = _BuffSize;
    return seg;
}

void SegmentPool::Return(FSegment& seg)
{
    // 풀에서 빌려온 메모리가 아니면 직접 해제
    if (!seg._IsPooled)
    {
        delete[] seg._Data;
        return;
    }

    // 풀에서 빌려온 메모리를 다시 풀로 되돌려준다
    _Buckets.push(seg);
}

2

3

MemoryPool 구조

크기 매핑 — ceil(log2(size)) 방식
IndexFromSize(size)가 ceil(log2(size)) - 4로 풀 인덱스를 계산:

20B → idx=1 → 32B 풀 (2^5)  |  40B → idx=2 → 64B 풀 (2^6)  |  100B → idx=3 → 128B 풀 (2^7)
512B 초과 → Find()가 nullptr → new byte[] 직접 할당 (IsPooled = false)

MemoryPool 핵심 코드

C++ MemoryPool 클래스 MemoryPool.h
class MemoryPool : public Singleton<MemoryPool>
{
    friend Singleton<MemoryPool>;

private:
    std::vector<class SegmentPool*> _Pools;
    const int _DefaultPow = 4;

public:
    virtual void Init() override;
    virtual void Destroy() override;

    FSegment Rent(int size);          // 요청 크기에 맞는 메모리를 빌려준다
    void Return(FSegment& seg);       // 대여한 메모리를 반납

private:
    int IndexFromSize(int size);      // 크기 → 풀 인덱스 변환
    SegmentPool* Find(int idx);       // 인덱스로 풀 찾기
};
C++ Init — 크기별 풀 생성 MemoryPool.cpp
void MemoryPool::Init()
{
    int pow = _DefaultPow;
    _Pools.push_back(new SegmentPool(static_cast<int>(std::pow(2, pow++)), 100)); // 16B x 100
    _Pools.push_back(new SegmentPool(static_cast<int>(std::pow(2, pow++)), 100)); // 32B x 100
    _Pools.push_back(new SegmentPool(static_cast<int>(std::pow(2, pow++)),  50)); // 64B x 50
    _Pools.push_back(new SegmentPool(static_cast<int>(std::pow(2, pow++)),  50)); // 128B x 50
    _Pools.push_back(new SegmentPool(static_cast<int>(std::pow(2, pow++)),  10)); // 256B x 10
    _Pools.push_back(new SegmentPool(static_cast<int>(std::pow(2, pow++)),  10)); // 512B x 10
}
C++ Rent — 크기에 맞는 풀 찾기 MemoryPool.cpp
FSegment MemoryPool::Rent(int size)
{
    int index = IndexFromSize(size);
    SegmentPool* pool = Find(index);
    if (!pool)
    {
        // 풀 범위 밖 → 직접 동적할당
        FSegment seg;
        seg._Data = new byte[size];
        seg._IsPooled = false;
        seg._Size = size;
        return seg;
    }

    return pool->Rent();
}

int MemoryPool::IndexFromSize(int size)
{
    int log = static_cast<int>(std::ceil(std::log2(size)));
    int idx = log - _DefaultPow;
    return idx;
}
IndexFromSize — 2의 거듭제곱 매핑
요청 크기를 ceil(log2(size))로 변환해서 풀 인덱스를 결정한다.
예: 40B → ceil(log2(40)) = 6 → 6 - 4 = 2 → 64B 풀 (_Pools[2])
풀 범위(16~512B)를 벗어나면 Find()가 nullptr을 반환 → 직접 new byte[] 할당.

ObjectPool 변경점

Day 01에서 만든 ObjectPool에 Init()과 Destroy()를 추가했다. Rent할 때 Init()으로 초기화하고, Return할 때 Destroy()로 정리한다. 생성자/소멸자 대신 이 함수들을 쓰는 이유는 — 객체를 파괴하지 않고 재활용하기 때문이다.

C++ ObjectPool::Rent — 템플릿 ObjectPool.h
template<typename T>
T* Rent(eObjectType type)
{
    auto pool = _Pools.find(type);
    if (pool == _Pools.end())
        return nullptr;

    T* retVal = nullptr;
    if (pool->second.empty())
    {
        retVal = new T();           // 풀이 비면 새로 생성
        return retVal;
    }
    else
    {
        retVal = dynamic_cast<T*>(pool->second.front());
        if (nullptr == retVal)
            return nullptr;

        pool->second.pop();
    }

    return retVal;
}
C++ ObjectPool::Return ObjectPool.cpp
void ObjectPool::Return(Object* obj)
{
    if (nullptr == obj)
        return;

    if (false == obj->_IsPooled)
    {
        delete obj;           // 풀에서 안 온 객체는 직접 해제
        return;
    }

    eObjectType type = obj->GetType();
    obj->SetActive(false);

    _Pools[type].push(obj);  // 풀에 다시 넣기
}
Init / Destroy vs 생성자 / 소멸자
생성자/소멸자는 객체의 탄생과 죽음에 한 번씩 불린다.
Init/Destroy는 객체를 재활용할 때마다 불린다.

ObjectPool에서 객체는 한 번 태어나면 프로그램 종료까지 안 죽는다. 대신 Rent할 때마다 Init으로 "리셋"하고, Return할 때마다 Destroy로 "정리"한다.

조건부 컴파일 전환

3가지 전략을 #ifdef로 전환할 수 있게 만들었다. Info.h에 매크로를 정의하면 해당 전략이 활성화된다.

C++ 조건부 컴파일 + 타입 앨리어스 Info.h
#ifdef _USE_MEMORY_POOL
using TestObjectPtr = class TestObject*;
using WeakTestObject = class TestObject*;
#elif _USE_OBJECT_POOL
using TestObjectPtr = class TestObject*;
using WeakTestObject = class TestObject*;
#else
using TestObjectPtr = std::shared_ptr<class TestObject>;
using WeakTestObject = std::weak_ptr<class TestObject>;
#endif
C++ Object.h — 3가지 버전 (핵심 부분만) Object.h
#ifdef _USE_MEMORY_POOL
#include "MemoryPool.h"

class Object
{
protected:
    FSegment _Segment;              // 반납할 때 필요한 정보

public:
    void* operator new(size_t size)
    {
        FSegment seg = MemoryPool::Instance().Rent((int)size);
        Object* obj = (Object*)seg._Data;
        obj->_Segment = seg;
        return seg._Data;
    }

    void operator delete(void* ptr) noexcept
    {
        Object* obj = (Object*)ptr;
        MemoryPool::Instance().Return(obj->_Segment);
    }

    virtual void Init() = 0;
    virtual void Destroy() = 0;
};

#elif _USE_OBJECT_POOL

class Object
{
protected:
    bool _IsPooled = false;
    eObjectType _Type = eObjectType::END;
    // ... Active, Enable 등

    virtual void Init() = 0;
    virtual void Destroy() = 0;
};

#else

class Object : public std::enable_shared_from_this<Object>
{
public:
    template<typename T>
    std::shared_ptr<T> This()
    {
        return std::dynamic_pointer_cast<T>(shared_from_this());
    }

    virtual void Init() = 0;
    virtual void Destroy() = 0;
};
#endif
개선 제안 — 템플릿 앨리어스로 보일러플레이트 제거
현재 구조는 클래스마다 using 6줄이 필요하지만, 템플릿 앨리어스를 쓰면 한 번 정의로 끝:
C++ template alias 대안
#if defined(_USE_MEMORY_POOL) || defined(_USE_OBJECT_POOL)
template<typename T> using Ptr = T*;
template<typename T> using WeakRef = T*;
#else
template<typename T> using Ptr = std::shared_ptr<T>;
template<typename T> using WeakRef = std::weak_ptr<T>;
#endif

TODO — 아직 구현 안 됨
  • Thread Safety — 현재 SegmentPool은 싱글스레드 전용. 멀티스레드 환경에서는 mutex나 lock-free queue가 필요.
  • 메모리 정렬(Alignment) — 현재는 정렬을 고려하지 않음. SIMD를 쓸 경우 16바이트 정렬이 필요할 수 있음.
  • 메모리 누수 추적 — Rent한 버킷을 Return하지 않으면 추적할 방법이 없음. 디버그용 카운터 추가 필요.
  • 풀 크기 자동 조절 — 현재는 고정 크기. 런타임에 사용 패턴을 분석해서 풀 크기를 조절하는 기능 미구현.
  • 512B 초과 객체 관리 — 큰 객체는 직접 malloc하는데, 이것도 별도 관리가 필요할 수 있음.
